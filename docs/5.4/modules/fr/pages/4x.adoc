==== Détails du comportement de XFF-FORWARDED-FOR

Dans un cluster Kubernetes, une application peut être exposée à l’extérieur du cluster par un NodePort, un LoadBalancer ou des services Ingress. Ces services remplacent généralement l'adresse IP source lors de l'exécution du NAT source (SNAT) sur les paquets. Comme l’adresse IP source d’origine est masquée, cela empêche {product-name} de reconnaître que la connexion provient en réalité de l’« externe ».

Pour préserver l'adresse IP source d'origine, l'utilisateur doit ajouter la ligne suivante aux services exposés, dans la section 'spec' de l'équilibreur de charge ou du contrôleur d'entrée. (Réf{nbsp}: https://kubernetes.io/docs/tutorials/services/source-ip/)

"externalTrafficPolicy":"Local"

De nombreuses implémentations de services LoadBalancer et de contrôleurs d'entrée ajouteront la ligne X-FORWARDED-FOR à l'en-tête de la requête HTTP pour communiquer la véritable adresse IP source aux applications backend. Dans la version 4.1.0, nous avons ajouté une fonctionnalité pour reconnaître cet ensemble d'en-têtes HTTP, identifier l'adresse IP source d'origine et appliquer la politique en conséquence.

Cette amélioration a créé des problèmes inattendus dans certaines configurations. Si la ligne ci-dessus a été ajoutée aux services exposés et que {product-name} les politiques réseau ont été créées de manière à s'attendre à ce que les connexions réseau proviennent de services proxy/ingress internes, car nous identifions maintenant que les connexions proviennent de l'"externe" au cluster, le trafic normal des applications pourrait déclencher des alertes ou être bloqué si les applications sont mises en mode "Protéger".

Dans la version 4.1.2, un commutateur a été ajouté pour désactiver cette fonctionnalité. La désactivation indique à {product-name} de ne pas identifier que la connexion provient de l'"externe" en utilisant les en-têtes X-FORWARDED-FOR. Par défaut, cela est activé, et l'en-tête X-FORWARDED-FOR est utilisé dans l'application des politiques. Pour le désactiver, allez dans Paramètres -> Configuration, et désactivez le paramètre "Correspondance de politique basée sur X-Forwarded-For".

==== Contrôle d'admission pour le partage de pods

. HostIPC - Contrôle si les conteneurs du pod peuvent partager l'espace de noms IPC de l'hôte.
. HostNetwork - Contrôle si le pod peut utiliser l'espace de noms réseau du nœud. Cela donne au pod accès au périphérique de boucle, aux services écoutant sur localhost, et pourrait être utilisé pour espionner l'activité réseau d'autres pods sur le même nœud.
